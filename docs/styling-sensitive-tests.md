# Styling-Sensitive Unit Tests

Unit tests that may be affected when theme overrides, brand-color changes, or `!important` removal land in Phase 1.

| Test file | Why it's styling-sensitive | What to watch when theme overrides land |
|-----------|---------------------------|----------------------------------------|
| `src/components/Pagination/Pagination.spec.tsx` | Wraps `<PaginationOutlined>` in a `createTheme()` / `ThemeProvider` to supply MUI tokens. If the default theme palette changes (e.g. `primary.main` moves to `#667eea`), pagination button colours and `aria-current` rendering could shift. | Re-verify buttons still render with correct `aria-current` after any custom theme is applied at the test level. |
| `src/components/SupplyDetailCard/SupplyDetailCard.spec.tsx` | Mocks MUI's `useMediaQuery` to simulate mobile (`true`) vs desktop (`false`) breakpoints. Assertions on `toBeVisible()` / `not.toBeVisible()` depend on the card toggling between a `<DropdownCard>` (mobile) and a flat card (desktop) based on `useMediaQuery`. | Any change to the MUI breakpoint used inside `SupplyDetailCard` (currently the default `theme.breakpoints`) will alter when the mock threshold fires, breaking the visibility assertions. |
| `src/components/DropdownCard/DropdownCard.spec.tsx` | Uses `not.toBeVisible()` to assert that collapsed content is hidden. MUI's `Collapse` hides via `visibility:hidden` / `display:none`; `toBeVisible()` checks computed visibility. If CSS specificity changes (e.g. `!important` removals) cause `Collapse` to no longer hide content, tests will false-pass. | After removing `!important` from transitions or display rules, confirm collapsed cards still read `not.toBeVisible()`. |
| `src/components/Menu/SideMenu.spec.tsx` | Asserts `toBeVisible()` when menu is open and `not.toBeVisible()` when closed, relying on MUI `Drawer`'s internal show/hide mechanism. | If theme overrides change `Drawer` variant or `sx` padding values, confirm menu items still toggle visibility correctly. |
| `src/components/Menu/ProfileMenu.spec.tsx` | Same pattern as `SideMenu` — collapsed state asserted via `not.toBeVisible()`. | Same risk as above. |
| `src/components/CardGrid/CardGrid.spec.tsx` | Queries MUI internal class `.MuiBox-root` to count rendered elements, and tests pass responsive `gap: { xs, sm }` / `columns: { xs, sm, md, lg }` props as `sx` values. Tests are labelled "trust MUI handles it" so they don't assert computed styles — but they would break if `CardGrid` stops accepting `sx`-style props. | Verify the `gap` and `columns` props still thread through `sx` after any Grid/Box refactor. |
| `src/components/CardGrid/LoadingCardGrid.spec.tsx` | Queries `.MuiPaper-root` and `.MuiSkeleton-rectangular` / `.MuiSkeleton-text` class names to count skeleton cards. These class names are MUI's internal generated names and will change if the MUI major version bumps or if skeleton markup is restructured. | After any MUI upgrade or skeleton `sx` changes, recount classes and update selectors. |

## Notes

- Tests using `toBeVisible()` rely on CSS properties (`display`, `visibility`, `opacity`) being computed correctly in jsdom. jsdom does not run a full CSS engine; it trusts component-level inline styles and CSSOM properties set directly. Brand-color changes (hex values) will **not** affect these tests. Structural changes to how MUI hides/shows elements (e.g. `display: none` → `opacity: 0`) **will** affect them.
- `Pagination.spec.tsx` is the only test that calls `createTheme()` explicitly. If a custom theme is later applied globally in tests (e.g. via a shared `renderWithTheme` helper), this test's wrapper theme may conflict with or override it.
- No spec currently asserts on computed color values, so the `#667eea` → CSS-variable migration in Phase 1 should not break any of the 28 tests directly. The risk is indirect: MUI components whose default `sx` used `palette.primary` will pick up the new primary colour, which could shift visual layout if it changes element sizing.
